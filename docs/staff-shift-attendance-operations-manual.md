# スタッフ管理（シフト/勤務）運用手順書

Task ID: `TASK-450`  
Version: `v1.4`（2026-04-18）

## 1. 対象機能
- 画面:
  - `/staffs`
  - `/attendance-punch`
- `/staffs` タブ:
  - `list` スタッフ一覧
  - `shift-settings` シフト設定
  - `shift` シフト管理
  - `shift-list` シフト一覧
  - `shift-history` シフト変更履歴（owner/admin）
  - `attendance-records` 勤怠管理
- サイドバー独立メニュー:
  - `勤怠打刻`（`/attendance-punch`）

## 2. 事前準備
- DBへ `supabase/supabase_staff_shift_management.sql` を適用する。
- 店舗ユーザーに `store_memberships` が設定されていることを確認する。
- 権限確認:
  - `owner/admin`: シフト・勤務の全管理操作
  - `staff`: 自分の打刻/申請と自分の実績閲覧

## 3. 日次運用フロー
1. 開店前に `shift` タブで当日シフトと不足警告を確認する。
2. 必要に応じて「指名予約と同期」「自動生成（下書き）」を実行する。
3. `draft` シフトを確認し、問題なければ公開する。
4. 当日はサイドバーの `勤怠打刻`（`/attendance-punch`）でスタッフが勤務打刻する。
5. 打刻漏れはスタッフが修正申請し、`owner/admin` が承認または却下する。
6. `attendance-records` タブで `incomplete` / `needs_review` と申請滞留を確認する。
7. 月末は `attendance-records` で月次確定し、締め後は閲覧専用で運用する。

## 4. 画面別操作

## 4.1 `shift`
- 手動追加: スタッフ/日付/開始/終了/休憩を入力して保存。
- 指名同期: 指名予約を基準に不足シフト候補を作成。
- 自動生成: 期間を指定して下書きシフトを作成。
  - 生成期間は `1-90日`（最大3か月）。店舗設定の「生成期間（日）」が上限。
  - 生成戦略:
    - `ルールベース`: 既存の制約（定休日/勤務可能枠/希望休/週上限）を満たす標準生成。
    - `最適化`: プロプラン向け。重み設定に基づいて候補の総合スコアを算出して生成。
  - `manual` の既存シフトは自動更新しない（保護）。
  - `auto/nomination_sync` は再生成時に更新・削除対象。
- 公開: `draft` を `published` へ遷移。
- 実行結果表示:
  - 生成件数・反映件数（作成/更新/削除/手動保護/制約警告）
  - 戦略（ルールベース/最適化）
  - 最適化実行時の総合スコア・代替案件数

## 4.2 `shift-settings`
- 店舗設定:
  - タイムゾーン
  - 生成期間（日）
  - 遅刻・早退猶予
  - 定休日（曜日CSV / 日付改行）
  - 生成優先ポリシー
- スタッフ設定:
  - 雇用区分
  - 週上限分
  - 連勤上限
  - 希望勤務分
  - 指名対応可否
  - 勤務可能枠（`曜日:開始-終了`）
  - 希望休（日付改行）
- プロ向け設定:
  - 最適化シフト生成ON/OFF
  - 定期自動運転ON/OFF
  - 重み設定（公平性/希望勤務達成/予約カバー/負荷平準化）
  - 定期自動運転ジョブ（daily/weekly、実行時刻、曜日、対象期間）

### 4.2.1 店舗設定の考え方
- タイムゾーン:
  - 原則は店舗の営業所在地に合わせる。
  - 打刻日付判定と日次集計の基準になるため、途中変更は月初に実施する。
- 生成期間（日）:
  - 何日先まで自動生成するか。`1-90日` の範囲で指定する。
  - まずは `14日` を推奨。
  - 変更が多い店舗は短め（7-14日）、予約が早く埋まる店舗は長め（21-28日）。
- 遅刻猶予（分）:
  - 「遅刻候補」と判定するまでの緩衝時間。
  - 運用例: 朝礼や準備時間を含めるなら 10-15分、厳格運用なら 5分。
- 早退猶予（分）:
  - 予定終了より何分早い退勤まで許容するか。
  - 片付け・引き継ぎを厳密に見る場合は短く（5-10分）設定。
- 定休日（曜日CSV）:
  - 毎週固定で休む曜日を設定する（`0=日, 6=土`）。
  - 例: 毎週火曜休みなら `2`。
- 定休日（日付改行）:
  - 祝日・年末年始・臨時休業など、単発の休業日を `YYYY-MM-DD` で列挙。
  - 例:
    - `2026-12-31`
    - `2027-01-01`
- 生成優先ポリシー:
  - `nomination_first`: 指名予約カバー最優先（通常はこれを推奨）。
  - `cost_first`: 人件費抑制を優先（繁閑差が大きい店舗向け）。
  - `fairness_first`: 勤務偏りの平準化を優先（複数スタッフ運用向け）。

### 4.2.2 スタッフ設定の考え方
- 雇用区分:
  - 管理用の区分。自動生成時の上限制御や運用判断の基準に使う。
  - 実態と一致させる（正社員/パート/アルバイト）。
- 週上限分:
  - 1週間で割り当てる勤務時間の上限。
  - 契約条件と労務方針に合わせる（例: 30時間なら `1800`）。
- 連勤上限:
  - 連続勤務可能日数の上限。
  - 体調管理を重視する場合は低め（4-5日）に設定。
- 希望勤務分:
  - 週あたりの目標勤務分（公平配分の目安）。
  - 正社員は高め、短時間勤務者は低めで設定する。
- 指名対応可否:
  - 指名予約の割当対象にするかを制御する。
  - 育成中スタッフや非指名運用スタッフは OFF にする。
- 勤務可能枠（`曜日:開始-終了`）:
  - 実際に勤務できる時間帯だけを登録する。
  - 例:
    - `1:09:00-18:00`（月曜 9-18時）
    - `5:10:00-16:00`（金曜 10-16時）
- 希望休（日付改行）:
  - 日単位で自動生成対象外にする日を設定する。
  - 例:
    - `2026-05-03`
    - `2026-05-10`

### 4.2.3 初期設定の推奨手順
1. 店舗定休日と営業時間相当（勤務可能枠）を先に登録する。
2. 全スタッフの「雇用区分・週上限・連勤上限」を埋める。
3. 指名予約に対応できるスタッフのみ `指名対応可` を ON にする。
4. ポリシーは `nomination_first` で開始し、2-4週間運用してから調整する。
5. `shift` タブで不足警告の出方を確認し、必要なら上限や勤務枠を再調整する。

## 4.3 `attendance-punch`
- 勤務打刻:
  - 出勤打刻
  - 退勤打刻
  - 休憩開始
  - 休憩終了

## 4.4 `attendance-records`
- 修正申請:
  - 対象日
  - 修正イベント種別
  - 修正時刻
  - 理由（必須）
- 承認（`owner/admin`）:
  - 承認時のみ `attendance_events` へ反映
  - 却下時は申請状態のみ更新
- 月次確定:
  - `owner/admin` が対象月を確定
  - 確定解除は `owner` のみ
  - 確定月は閲覧専用（打刻/申請/承認は実行不可）

### 4.4.1 確定月（閲覧専用）運用
- 画面に `月次状態: 確定済み` と「この月は閲覧のみ」が表示される。
- 確定済み月で無効になる操作:
  - 打刻（出勤/退勤/休憩開始/休憩終了）
  - 休暇申請作成
  - 勤務修正申請作成
  - 休暇申請レビュー（承認/却下/差戻し）
  - 勤務修正申請レビュー（承認/却下）
- API では上記更新系操作は `409` で拒否される。
- 再編集が必要な場合は `owner` が「確定解除」を実行してから再操作する。

## 5. 運用ルール
- 打刻・申請は原則当日中に処理する。
- `needs_review` は翌営業日開始までに解消する。
- `pending` 申請は1営業日以内に承認/却下する。
- 指名予約不足警告は営業前にゼロ化を目標とする。
- 自動生成後は差分件数（作成/更新/削除/手動スキップ）を確認してから公開する。

## 6. 既知の制約（v1）
- カレンダーは週/日中心。月ビューは次フェーズ対応。
- 指名予約連動の対象予約ステータスは `confirmed/pending` のみ。
- 定期自動運転ジョブはプロプランのみ（スタンダード以下は手動実行のみ）。
- 修正申請の重複 `pending` 抑止はUI/運用中心（DB強制は未導入）。
- 打刻順序不整合は一部を `needs_review` で吸収し、管理者確認で確定する。
- 監査ログ専用テーブルは別タスクで拡張予定。

## 7. 障害時対応
- 打刻保存エラー時:
  - 同操作を1回再実行
  - 失敗継続時は修正申請で暫定記録
- 申請承認エラー時:
  - `pending` のまま残るため再承認を試行
  - 重複反映防止のため承認連打は避ける
- シフト生成結果不整合時:
  - 下書きを削除し、期間を絞って再生成
  - 必要なら手動追加へ切替

## 9. v1.4実装内容（完了）

### 9.1 ナビゲーション変更
- `勤務打刻` はスタッフ管理タブから分離し、サイドバーのメインカテゴリ `勤怠打刻` に移動する。
- 対象ロールは `owner/admin/staff` 全員。

### 9.2 画面構成（2段階）
- 画面1: スタッフ選択（`/attendance-punch`）
  - `staff`: 自分のみ選択可
  - `owner/admin`: 全スタッフ選択可
- 画面2: 個別打刻（`/attendance-punch?staff_id=...`）
  - `出勤打刻/退勤打刻/休憩開始/休憩終了` を実行

### 9.2.1 勤怠メニュー表示の店舗設定連動
- 店舗設定に `勤怠打刻を有効化` を追加する。
- 運用ルール:
  - ON時は `勤怠打刻` と `勤務実績` の両方がサイドバーメニューに表示される。
  - OFF時は両メニューとも非表示となり、直接URLアクセスしても利用できない。
  - 機能有効化後にメニュー表示されることを確認してから現場へ案内する。

### 9.3 位置情報不正防止
- 店舗設定に以下を追加する。
  - `位置情報を取得する`（チェックボックス）
  - `基準緯度(lat)` / `基準経度(lng)` / `許容半径(m)`（初期値200）
- 設定ON時は4イベントすべてで位置情報を必須にする。
- 位置情報拒否/失敗/半径外は打刻拒否とする。
- `owner/admin` の代理打刻時も同一ルールを適用する。

### 9.4 位置情報の記録と表示
- 記録項目:
  - `lat` / `lng` / `accuracy` / `captured_at` / `is_within_radius`
- 範囲外などの拒否ケースも監査目的で記録を残す。
- 位置情報の表示は `owner/admin` のみ。
- 勤務実績一覧では詳細表示（行展開）で確認する。
- 保持期間は無期限。

### 9.5 プロ向けシフト最適化・定期自動運転
- シフト設定に「プロ向けシフト最適化」カードを追加。
  - 最適化シフト生成ON/OFF
  - 定期自動運転ON/OFF
  - 重み設定（合計1.0必須）
- シフト設定に「定期自動運転ジョブ」カードを追加。
  - ジョブ追加（daily/weekly、実行時刻、曜日、対象期間）
  - ジョブ編集/有効無効/削除
  - 実行履歴確認（成功/失敗、対象期間、生成件数）
- シフト管理の自動生成に戦略セレクタを追加。
  - `ルールベース` / `最適化`
  - 実行結果に戦略と総合スコアを表示
- シフト変更履歴に戦略・総合スコア列を追加。

## 10. プラン別運用境界（合意）

### 10.1 ライト
- 利用可能:
  - 勤怠打刻
  - 勤務実績表示
- 表示範囲:
  - `staff`: 本人のみ
  - `owner/admin`: 全体閲覧可
- 利用不可:
  - シフト管理
  - 自動生成

### 10.2 スタンダード
- ライトに加えて利用可能:
  - シフト手動管理（作成/編集/公開）
  - ルールベース自動生成（予約考慮、手動実行）
- 画面上の扱い:
  - 最適化チェックボックスは表示するが無効化（プロ誘導）

### 10.3 プロ
- スタンダードに加えて利用可能:
  - 定期自動運転（cron）
  - 運用自動化
  - シフト生成最適化（ON/OFF）
- 運用上の確認項目:
  - 最適化スコア
  - 生成理由
  - 例外時代替提案
