Task ID: `TASK-450`

実装状況サマリー:

- `v1.3`: 完了（`勤怠打刻` の独立メニュー化、店舗設定の `勤怠打刻を有効化` と `位置情報必須`、基準座標/半径判定、位置情報監査項目保存）

## 1. 目的
- 既存のスタッフ管理（`/staffs`）に対して、シフト管理（勤務予定）と勤務管理（勤務実績）を追加する。
- 予約運用・実績管理・労務確認の基盤を分離し、現場の「予定」と「実績」の差分を可視化できる状態にする。

## 2. 用語定義
- シフト: 事前に作成する勤務予定（誰が、いつ、働く予定か）。
- 勤務実績: 実際の出勤・退勤・休憩などの記録。
- 打刻: 出勤/退勤/休憩開始/休憩終了の実績入力操作。
- 修正申請: 打刻漏れや誤打刻に対する事後修正依頼。

## 3. 対象ロール
- `owner`: 全操作可（シフト作成、公開、実績修正承認、締め）。
- `admin`: シフト作成・編集、実績確認、修正申請の承認可。
- `staff`: 自身のシフト閲覧、自身の打刻、自身の修正申請のみ可。

## 4. スコープ（Step 1確定）
1. シフト管理（予定）
- 日別/週別のシフト作成、編集、削除
- スタッフ別の勤務予定一覧
- シフト公開状態（下書き/公開）の管理

2. 勤務管理（実績）
- 出勤/退勤/休憩開始/休憩終了の打刻
- 打刻履歴の閲覧（本人・管理者）
- 打刻修正申請と承認/却下

3. 運用補助
- 遅刻/早退/未打刻の検知フラグ
- 日跨ぎ勤務（深夜帯）の扱い
- 月次締め前の差分確認（予定対実績）

4. 予約連動・自動シフト補助
- 指名あり予約とシフト予定の連動
- 指名予約が存在する未来日に対するシフト不足警告と反映支援
- ルールベースの自動シフト生成（提案）
- シフト生成に必要な設定UI（店舗条件・スタッフ条件・ポリシー）

## 5. スコープ外（後続フェーズ）
- 自動給与計算/給与システム連携
- 生体認証や端末固有認証付き打刻
- 複数店舗跨ぎの勤務最適化自動提案
- 外部カレンダー双方向同期

## 6. 機能要件

### FR-01 シフト作成・編集
- `owner/admin` は店舗内スタッフに対してシフトを作成・更新できる。
- シフトは `勤務日`、`開始時刻`、`終了時刻`、`休憩予定分`、`メモ` を保持する。
- 下書き保存中は `staff` へ公開されない。

### FR-02 シフト公開
- シフトは `draft` から `published` へ遷移できる。
- 公開後は `staff` が自分の予定として閲覧できる。

### FR-03 打刻
- `staff` は自身の勤務に対して出勤/退勤/休憩開始/休憩終了を記録できる。
- 不正順序（退勤前の出勤なし等）はバリデーションで拒否する。

### FR-04 勤務実績算出
- 実働時間は「退勤 - 出勤 - 休憩合計」で算出する。
- 日跨ぎ時は業務日をシフト基準日へ紐づけて集計する。

### FR-05 例外検知
- シフト開始から一定猶予後も出勤打刻がない場合に `未打刻/遅刻候補` とする。
- 予定終了より早い退勤は `早退候補` フラグを付与する。

### FR-06 修正申請
- `staff` は自身の勤務実績に対して修正申請を提出できる。
- `owner/admin` は申請を承認/却下し、承認時のみ実績へ反映する。

### FR-07 監査ログ
- シフト作成・更新・公開、打刻、修正申請、承認/却下を監査可能な形で記録する。
- 監査ログには `actor_user_id`、`store_id`、操作種別、変更前後を保持する。

### FR-08 マルチストア/RLS
- データは常に `store_id` でスコープされる。
- 他店舗スタッフのシフト/実績参照や更新は拒否する。

### FR-09 指名予約連動
- 指名あり予約（`appointments.staff_id` が設定済み）をシフト計画へ取り込む。
- 対象日時に該当スタッフのシフトが未登録の場合、`勤務必須候補` を自動付与する。
- 予約変更/キャンセル時は差分再計算し、既存シフトとの差異を通知する。

### FR-10 未来予約日の不足警告
- 将来日に指名あり予約があるのにシフト未配置の枠が存在する場合、警告を表示する。
- 管理者は警告行からワンクリックで候補シフトを作成できる。

### FR-11 自動シフト生成（段階導入）
- Phase 1（必須）: ルールベース自動生成
  - 指名予約を固定制約として先に配置する。
  - 店舗定休日・営業時間・スタッフ勤務可能時間を制約として適用する。
  - 最低人員などの店舗ポリシーを満たすよう不足枠を補完する。
- Phase 2（拡張）: スコア最適化
  - 公平性、残業抑制、希望休尊重、稼働率をスコア化し最適案を提示する。
- 自動生成結果は公開前にプレビュー/手修正できることを必須とする。

### FR-13 自動生成期間（v1.1）
- 自動生成期間は `1-90日` を設定可能とする（最大3か月）。
- 管理画面で設定した期間を上限として、自動生成の対象範囲を決定する。

### FR-14 既存シフト保護と差分履歴（v1.1）
- `source_type=manual` のシフトは自動生成で上書きしない。
- `source_type=auto|nomination_sync` は再生成時の更新対象とする。
- 自動生成実行ごとに `run_id` を採番し、`created/updated/deleted/skipped_manual` の差分履歴を保持する。
- 変更が発生したシフトはUI上で変更有無を識別できること。

### FR-15 指名予約優先の運用定義（v1.1）
- 対象予約は `confirmed/pending` のみとする。
- 指名予約がある日は対象スタッフを「その日出勤必須」とする。
- 予約時刻の完全包含は必須としない。未包含は警告（不足）として記録する。
- 定休日と指名予約が衝突した場合は自動営業化せず、警告のみ表示する。

### FR-16 生成制約（v1.1）
- 1日あたりスタッフ1本のシフトのみ生成する（中抜けなし）。
- 雇用区分・週上限・連勤上限は厳格適用する。
- 制約違反時は当該スタッフを未配置として警告を作成し、全体処理は継続する（全体失敗にしない）。
- 休憩時間は固定値（既存 `planned_break_minutes`）を用い、自動補正は行わない。

### FR-17 希望休（日単位）対応（v1.1）
- 希望休は日単位で設定できること（時間帯指定は次フェーズ）。
- 希望休日は自動生成の配置対象から除外する。

### FR-18 実行タイミング（v1.1）
- v1.1では管理者の手動実行のみ対応する。
- 将来の cron 定期実行を見据えた実行履歴/冪等性の設計とする。

### FR-12 シフト生成設定UI
- 店舗設定（シフト用）
  - 定休日: 曜日指定（例: 毎週火曜）と日付指定（例: 2026-12-31）を両対応
  - 営業時間（曜日別）
  - 必要最低人員（時間帯別に拡張可能な設計）
- スタッフ設定（シフト用）
  - 雇用区分（正社員/パート/アルバイト）
  - 勤務可能曜日・時間帯
  - 週あたり勤務上限時間
  - 連勤上限日数
  - 希望休
  - 指名対応可否/優先度
- ポリシー設定
  - 生成優先軸（指名予約優先/コスト優先/公平性優先）
  - 猶予設定（遅刻判定分、シフト自動反映時のバッファ）

## 7. 非機能要件
- 性能: 日別シフト一覧・日別勤務実績一覧は P95 2秒以内を目標とする。
- 可用性: 打刻失敗時に再実行でき、重複登録を抑止できること。
- 監査性: 実績変更の経路（初回打刻/修正申請/承認）を追跡可能であること。
- セキュリティ: `staff` が他者データを操作できないこと。

## 8. 業務フロー（要件確定）
1. `owner/admin` が週次シフトを下書き作成する。
2. レビュー後に公開し、`staff` が閲覧する。
3. 当日、`staff` が出勤・休憩・退勤を打刻する。
4. 打刻漏れや誤りがある場合、`staff` が修正申請する。
5. `owner/admin` が承認/却下し、月次締め前に予定対実績を確認する。
6. 未来予約に指名あり予約が追加/変更された場合、シフト差分を再計算して不足警告を表示する。
7. 管理者は必要に応じて自動生成を実行し、提案シフトを確認・調整して公開する。

## 9. 例外ケース
- 遅刻: 出勤打刻が予定開始を超過した場合に遅刻候補化。
- 早退: 退勤打刻が予定終了より前の場合に早退候補化。
- 打刻漏れ: 出勤/退勤が片方欠落している状態を未完了勤務として扱う。
- 日跨ぎ: 22:00-翌06:00等の勤務を単一勤務として保持する。
- 修正競合: 承認処理中に新規修正申請が来た場合は最新申請を優先し、旧申請は再確認状態にする。

## 10. 受け入れ基準（Given/When/Then）
1. シフト公開
- Given `owner` が下書きシフトを持つ
- When 公開操作を行う
- Then 対象 `staff` に公開状態で表示される

2. 正常打刻
- Given 当日の公開済みシフトがある
- When `staff` が出勤→休憩開始→休憩終了→退勤を順に打刻する
- Then 勤務実績が完了し実働時間が算出される

3. 打刻順序エラー
- Given 出勤打刻が未実施
- When `staff` が退勤打刻を実行する
- Then バリデーションエラーで拒否される

4. 他者データ操作防止
- Given `staff` が他スタッフIDを指定する
- When 打刻または修正申請を実行する
- Then 403または404で拒否される

5. 修正申請承認
- Given 打刻漏れの勤務実績がある
- When `staff` が修正申請し `admin` が承認する
- Then 申請内容が勤務実績に反映され監査ログが残る

6. 日跨ぎ勤務
- Given 22:00-翌06:00のシフトがある
- When 正常に打刻が完了する
- Then 単一勤務として実績が保存される

7. 指名予約連動
- Given 未来日の指名あり予約が登録されている
- When 対象スタッフの同時間帯シフトが未登録である
- Then システムは不足警告を表示し、勤務必須候補を提案する

8. 自動シフト生成
- Given 店舗定休日・営業時間・スタッフ勤務条件が設定済みである
- When 管理者が自動生成を実行する
- Then 指名予約を満たし制約違反のないシフト案が生成される

## 14. 勤務管理拡張要件（v1.2提案）

### 14.1 目的
- 現行の「打刻・実績表示」に加えて、遅刻/早退/欠勤/休暇/承認/締めの業務を一貫管理できる状態にする。
- 現場運用（申請・承認）と労務運用（月次確定・監査）を同一基盤で扱えるようにする。

### 14.2 管理対象ステータス
- 勤務日ステータスとして以下を扱う。
  - `normal`（通常）
  - `late`（遅刻）
  - `early_leave`（早退）
  - `absence`（欠勤）
  - `paid_leave`（有休）
  - `half_leave_am`（半休: 午前）
  - `half_leave_pm`（半休: 午後）
  - `special_leave`（特別休暇）
  - `compensatory_leave`（代休）
  - `holiday_work`（休日出勤）
  - `missing_punch`（打刻漏れ）
  - `needs_review`（要確認）

### 14.3 機能要件（FR-19〜FR-28）

#### FR-19 日次判定（遅刻・早退・欠勤・残業）
- シフト予定と打刻実績の差分から日次判定を自動算出する。
- 遅刻/早退の判定は `store_shift_settings.late_grace_minutes` / `early_leave_grace_minutes` を使用する。
- 予定勤務日に出勤打刻が存在しない場合は `absence` または `missing_punch` 候補とする（運用設定で切替可能）。
- 残業時間は「実勤務分 - 所定勤務分」で算出し、月次集計へ反映する。

#### FR-20 休暇申請（有休・欠勤・特休・半休）
- `staff` は休暇申請を作成できる。
- 申請種別は `paid_leave / half_leave_am / half_leave_pm / special_leave / absence` を最低対応とする。
- `owner/admin` は承認/却下/差戻しができる。
- 承認済み申請は日次実績へ自動反映し、打刻不足フラグと競合しないよう整合を取る。

#### FR-21 有休残数管理
- スタッフごとに有休残数（付与/取得/失効/繰越）を管理する。
- 休暇申請承認時に残数を自動減算する。
- 取消/却下/差戻し時は残数を自動復元する。

#### FR-22 修正申請ワークフロー拡張
- 既存の打刻修正申請に `差戻し` と `再申請` を追加する。
- 申請には理由必須、任意で添付メモを保持できる。
- 承認時は反映前後の差分を監査ログへ保存する。

#### FR-23 勤務実績（月次）
- 「勤務実績」画面は月単位表示を基本とし、対象月の全日を表示する。
- 日別列として、`出勤/退勤/休憩開始/休憩終了/休憩分/勤務分/状態/申請状態` を表示する。
- 月次サマリとして、`出勤日数/遅刻回数/早退回数/欠勤日数/有休取得日数/残業分` を表示する。

#### FR-24 締め処理（勤怠確定）
- 管理者は月次で勤怠を確定できる。
- 確定後は対象月を閲覧専用にし、打刻・休暇申請・修正申請・申請承認/却下/差戻しを禁止する。
- 再オープン権限は `owner` のみとする。

#### FR-25 アラート管理
- 以下のアラートを管理画面で一覧表示する。
  - 未承認申請の滞留
  - 連勤上限超過
  - 月残業しきい値超過
  - 欠勤/打刻漏れ未解消
- アラートは `info/warn/critical` の優先度を持つ。

#### FR-26 監査ログ拡張
- 休暇申請、承認、差戻し、勤怠確定、再オープンを監査対象に追加する。
- 監査ログは `actor_user_id`、対象スタッフ、対象日/月、操作種別、変更前後を保持する。

#### FR-27 権限制御
- `owner`: 全機能（設定・承認・確定・再オープン）
- `admin`: 申請承認・実績確認・修正反映（再オープン不可）
- `staff`: 自身の打刻、申請作成、実績閲覧のみ

#### FR-28 外部連携用エクスポート
- 月次勤務実績をCSV出力できる。
- 出力項目は給与計算連携を想定し、`日付/スタッフ/勤務分/残業分/休暇種別/状態` を含む。

### 14.4 非機能要件（NFR-05〜NFR-09）
- NFR-05 性能: 月次勤務実績（1スタッフ1か月）は P95 2秒以内で表示できること。
- NFR-06 一貫性: 申請承認と実績再計算は同一トランザクションで反映すること。
- NFR-07 監査性: 変更履歴は削除不可で保持し、追跡可能であること。
- NFR-08 可用性: 打刻・申請APIは冪等性キーを受け付け、二重送信を抑止すること。
- NFR-09 セキュリティ: 他店舗データおよび他スタッフデータへの越境アクセスをRLSで拒否すること。

### 14.5 データ項目（追加提案）

### 14.7 受け入れ基準（追加）

2. 有休承認

4. 再オープン権限
- Given 対象月が `closed`
- When `admin` が再オープンを実行する
- Then 403で拒否される
- And `owner` の実行時のみ `open` に戻る

## 15. 勤務打刻独立・位置情報要件（v1.3合意）

### 15.1 目的
- 勤務打刻をスタッフ管理配下のタブから分離し、日常オペレーションの入口を単純化する。
- 店舗設定に応じて位置情報を必須化し、なりすまし・遠隔打刻を抑止する。

### 15.2 機能要件（FR-29〜FR-34）

#### FR-29 ナビゲーション変更
- `勤務打刻`（表示名: `勤怠打刻`）をサイドバーのメインカテゴリとして追加する。
- スタッフ管理（`/staffs`）から勤務打刻タブは削除する。
- `owner/admin/staff` の全ロールでカテゴリ表示する。

#### FR-30 2段階打刻UI
- 画面1: 打刻対象スタッフ一覧（`staff` は自分のみ表示、`owner/admin` は全スタッフ表示）。
- 画面2: スタッフ個別打刻画面（`出勤打刻/退勤打刻/休憩開始/休憩終了`）。
- 遷移は `勤務打刻一覧 -> スタッフ個別` の画面遷移方式とする。

#### FR-31 代理打刻
- `owner/admin` は任意スタッフを選択して代理打刻できる。
- 代理打刻時も本人打刻時と同じバリデーションを適用する。

#### FR-32 位置情報必須設定
- 店舗設定に `位置情報を取得する` チェックボックスを追加する。
- 同設定ON時は、4イベント（出勤/退勤/休憩開始/休憩終了）すべてで位置情報取得を必須とする。
- 位置情報取得拒否・取得失敗時は打刻を拒否する。

#### FR-32.1 勤怠機能の店舗設定ON/OFF
- 店舗設定で `勤怠打刻を有効化` フラグを設定できること。
- `勤怠打刻を有効化=true` の場合、`勤怠打刻` と `勤務実績確認` の両方を有効化する。
- `勤怠打刻を有効化=false` の場合、`勤怠打刻` と `勤務実績確認` の両方を無効化する。
- メニュー表示は設定に連動し、ON時のみ表示する。
- OFF時は画面遷移/API実行ともに拒否される（UI非表示 + サーバ側ガード）。
- 本設定はロールに依らず同一店舗内で共通適用する。

#### FR-33 距離判定と不正防止
- 店舗設定に `基準緯度(lat)`・`基準経度(lng)`・`許容半径(m)` を追加する。
- 許容半径の初期値は `200m` とする。
- 打刻時に店舗基準座標からの距離判定を行い、範囲外は打刻拒否する。
- 範囲外/拒否時の記録は監査目的で保存する。

#### FR-34 位置情報監査表示
- 位置情報は `owner/admin` のみ閲覧可能とする。
- 勤務実績一覧では列追加ではなく「行の詳細表示」で確認できるようにする。
- 保持期間は無期限とする。

### 15.3 データ保持要件（位置情報）
- 保存項目:
  - `lat`（緯度）
  - `lng`（経度）
  - `accuracy`（推定精度メートル）
  - `captured_at`（端末取得時刻）
  - `is_within_radius`（許容半径内判定）

### 15.4 受け入れ基準（追加）
1. 位置情報必須OFF
- Given 店舗設定で位置情報取得がOFF
- When スタッフが勤務打刻する
- Then 従来通り打刻成功する

2. 位置情報必須ON（許可/範囲内）
- Given 位置情報取得ONかつ店舗半径200m内
- When 出勤打刻を実行する
- Then 打刻成功し、位置情報が保存される

3. 位置情報必須ON（拒否/失敗）
- Given 位置情報取得ON
- When 端末で位置情報を拒否または取得失敗する
- Then 打刻は拒否される

4. 位置情報必須ON（範囲外）
- Given 位置情報取得ONかつ店舗半径200m外
- When 勤務打刻を実行する
- Then 打刻は拒否され、拒否理由と位置情報記録が残る

## 16. プラン別提供範囲（合意）

### 16.1 ライト
- 勤怠打刻（出勤/退勤/休憩開始/休憩終了）
- 勤務実績表示
  - `staff`: 本人実績のみ
  - `owner/admin`: 全体閲覧可
- シフト管理（手動作成/公開）および自動生成は提供対象外

### 16.2 スタンダード
- ライトの全機能
- シフト管理（手動作成/編集/公開）
- ルールベース自動生成（予約考慮あり）
- 自動生成は手動実行のみ
- 最適化チェックボックスは表示するが無効化（アップセル導線）

### 16.3 プロ
- スタンダードの全機能
- 定期自動運転（cron）
- 運用自動化（通知/再実行など）
- シフト生成最適化（ON/OFFチェックボックス有効）
- 最低ライン機能:
  - 最適化スコア可視化
  - 生成理由表示
  - 例外時代替提案

## 17. プロ向けシフト最適化要件（v1.4）

### 17.1 目的
- スタンダードの「制約を満たす生成」に対して、プロでは「制約を満たした上で、より運用しやすい案」を自動提案する。
- 最適化の結果はブラックボックスにせず、管理者が採用判断できる説明情報を必須で表示する。

### 17.2 適用範囲
- 対象プラン: `pro` のみ。
- 実行方式:
  - 手動実行（即時生成）を必須。
  - 定期自動運転（毎日/毎週）を必須。
- 対象期間: `1-90日`（既存上限を継承）。

### 17.3 最適化の前提ルール
- 既存の固定制約（定休日、勤務可能枠、指名予約カバー、手動シフト保護、連勤上限、週上限）は最優先で維持する。
- 固定制約を破ってスコアを上げる生成は禁止する。
- 固定制約を満たせない場合は、無理に割当せず警告と代替案を返す。

### 17.4 評価軸（スコア）
#### FR-35 総合スコア算出
- 総合スコアは `0-100` で算出する。
- 構成要素:
  - 公平性スコア（勤務分数の偏り）
  - 希望勤務充足スコア（スタッフ別希望勤務分に対する充足）
  - 予約カバースコア（指名予約カバー率）
  - 労働負荷健全性スコア（連勤上限・週上限への近接度）
- 初期重み（合意値）:
  - 公平性 `0.35`
  - 希望勤務充足 `0.25`
  - 予約カバー `0.30`
  - 労働負荷健全性 `0.10`
- 重み合計は常に `1.0`。

#### FR-36 重みカスタマイズ
- `owner/admin` は店舗単位で重みを変更できる（プロのみ）。
- 重みは各項目 `0.0-1.0`、合計 `1.0` を必須とする。
- 未設定時は初期重みを適用する。

#### FR-37 最適化ON/OFF
- シフト設定に `最適化ON/OFF` を追加する（プロのみ有効）。
- OFF時は既存のルールベース生成（スタンダード相当）で動作する。
- ON時のみ最適化スコア計算・比較を実行する。

### 17.5 出力要件（説明可能性）
#### FR-38 生成理由表示
- 各生成実行ごとに以下を表示する:
  - 総合スコア
  - 各評価軸のスコア
  - 前回実行との差分（改善/悪化）
  - 主な判断理由（上位3件）

#### FR-39 変更影響表示
- 前回採用シフトとの差分を `created/updated/deleted` 件数で表示する。
- 影響が大きい更新（同一スタッフの開始時刻が60分以上変化など）はハイライト表示する。

#### FR-40 不足時の代替案
- 固定制約により未充足が残る場合、少なくとも2種類の代替案を返す:
  - 案A: 追加出勤候補（誰を追加すれば埋まるか）
  - 案B: 時間帯調整候補（どの枠を短縮/移動すると解消するか）
- 代替案は「推奨理由」と「影響（連勤/週上限への影響）」を含む。

### 17.6 定期自動運転要件
#### FR-41 スケジュール設定
- プロで `定期自動運転` を設定可能とする。
- 最低限の設定:
  - 実行頻度（毎日/毎週）
  - 実行時刻（店舗タイムゾーン）
  - 対象期間（先何日を生成するか）
  - 実行モード（draft作成のみ）

#### FR-42 実行結果通知
- 定期実行の成功/失敗を履歴に残す。
- 失敗時は `owner/admin` が原因を判別できるエラー要約を保持する。

### 17.7 セキュリティ・権限
#### FR-43 プラン/ロール制御
- `pro` 未満の店舗では最適化実行・重み編集・定期自動運転設定を `403` で拒否する。
- 実行/設定変更は `owner/admin` のみ許可する。

### 17.8 受け入れ基準（v1.4）
1. 最適化OFF
- Given プロ店舗で最適化OFF
- When 自動生成を実行
- Then ルールベース生成のみ実行され、スコア比較は実施されない

2. 最適化ON
- Given プロ店舗で最適化ON
- When 自動生成を実行
- Then 総合スコアと評価軸スコアが保存・表示される

3. 代替案表示
- Given 固定制約により未充足枠が残る
- When 自動生成を実行
- Then 追加出勤候補と時間帯調整候補の両方が表示される

4. プランガード
- Given スタンダード店舗
- When 最適化実行APIまたは定期自動運転設定APIを呼ぶ
- Then `403` で拒否される
