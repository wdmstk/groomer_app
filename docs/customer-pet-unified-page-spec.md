# 顧客・ペット統合管理ページ 仕様（確定版）

- Task ID: `TASK-438`
- 作成日: 2026-04-02
- ステータス: Confirmed（実装着手可）
- 対象: 店舗運用画面（スタッフ/管理者向け）

## 1. 背景と目的

現状、`/customers`（顧客管理）と`/pets`（ペット管理）が分離しており、1顧客に紐づく複数ペットを横断して見る際に画面遷移が発生する。

本タスクでは、既存ページは残したまま、別ページに「顧客起点で基本情報とペット情報をタブ管理できる統合画面」を新設する。

## 2. スコープ

### 2.1 実施すること（今回）

- 新規ページを追加する（既存`/customers`・`/pets`は削除しない）
- 顧客ページ内に以下タブを実装する
  - 基本情報タブ
  - ペット名ごとのタブ（複数）
- 表示情報は以下を満たす
  - 基本情報タブ: 現在の顧客管理一覧の情報をすべて表示
  - ペットタブ: 現在のペット一覧の情報をすべて表示
  - ペットタブ内に紐づくカルテ一覧・日誌一覧を表示

### 2.2 今回は実施しないこと

- 既存`/customers`・`/pets`の廃止やリダイレクト
- DBスキーマ変更
- 予約/会計/同意書機能の仕様変更

## 3. 画面・導線仕様（案）

## 3.1 新規ページ

- 最終ページ名（提案）: `顧客管理`
- 新規ページURL（最終想定）: `/customers/manage`
- β期間のサイドバー表示名: `顧客管理（β）`
- 既存「顧客管理」「ペット管理」は当面そのまま残す
- 将来は`/customers`を統合画面へ寄せる前提で、β期間は別URL運用とする

## 3.2 画面構成

- 上段: 顧客検索/選択エリア
  - 顧客名・電話番号で検索
  - 顧客を選択すると下段の詳細カードを表示
- 中段: 顧客タブ
  - タブ1: `基本情報`
  - タブ2以降: `{ペット名}`（顧客に紐づく件数分）
- 下段: タブ内容表示

## 3.3 基本情報タブ（顧客一覧項目を完全包含）

表示対象（既存`/customers?tab=list`準拠）:
- 氏名
- 電話番号
- メール
- 住所
- LINE連携状態（連携済み/未連携 + ID詳細）
- 来店経路
- LTVランク
- 年間売上
- 来店回数
- 平均単価
- オプション利用率
- タグ
- 会員証URL情報（有効期限/最終利用）
- 無断CXL件数バッジ

アクション:
- 顧客編集
- 空き枠待ち登録
- 顧客削除

## 3.4 ペット名タブ（ペット一覧項目を完全包含 + 連携一覧）

表示対象（既存`/pets?tab=list`準拠）:
- ペット名
- 飼い主
- 犬種
- 性別
- 生年月日
- 体重
- ワクチン接種日
- 持病
- 注意事項

追加表示:
- 紐づくカルテ一覧（`medical_records`）
  - 表示項目案: 記録日、メニュー、担当、ステータス、タグ
  - 各行からカルテ編集/詳細へ遷移
- 紐づく日誌一覧（`journal_entries` + `journal_entry_pets`）
  - 表示項目案: 投稿日時、公開範囲、本文先頭、写真/動画件数
  - 各行から日誌詳細（またはアルバム）へ遷移

アクション:
- ペット編集
- 日誌アルバムへ遷移
- ペット削除

## 3.5 空状態

- 顧客未選択時: 「左上から顧客を選択してください」
- 顧客にペットが0件: `基本情報`のみ表示し、ペット追加導線を表示
- カルテ0件/日誌0件: それぞれ空メッセージを表示

## 4. 非機能・実装方針

- 非破壊方針:
  - 既存API（`/api/customers*`, `/api/pets*`）は互換維持
  - 新ページは既存データ取得ロジックを再利用
- 性能:
  - 初回は基本情報のみ先読み
  - ペットタブ選択時にカルテ/日誌を遅延読込
- 店舗設定:
  - カルテ一覧件数`N`は店舗設定で可変（例: 5〜100）
  - 日誌一覧は店舗設定で「公開済みのみ / 下書き含む」を切替可能
  - 設定未指定時の初期値:
    - カルテ一覧件数`N`: `10`
    - 日誌表示: `公開済みのみ`
- 権限:
  - 既存`createStoreScopedClient`前提の店舗スコープを維持
- テスト:
  - 単体: presentation helper拡張分
  - E2E: 顧客選択 → タブ切替 → カルテ/日誌表示の主要導線

## 5. 想定される問題点・懸念点

- データ量増加時に「1画面で全部表示」が重くなる
- 同一画面に編集導線が増え、誤操作リスクが上がる
- 顧客とペットの双方削除導線が近接すると危険
- 既存ページと新ページの二重運用期間に、オペレーション手順が混在する

## 6. 改善案（UX/UI）

- 顧客選択時にヘッダー固定の「コンテキストバー」を表示
  - 例: 顧客名 / 連絡先 / ペット頭数 / 最終来店日
- ペットタブに補助情報バッジを表示
  - 例: `カルテ12件` `日誌4件` `要注意`
- 破壊操作の保護
  - 削除ボタンをドロップダウン内に格納
  - 「名前再入力で削除確定」モーダル採用
- モバイル最適化
  - タブを横スクロールチップ化
  - 重要アクション（編集/日誌/カルテ）を画面下固定アクションに集約
- 比較しやすさ向上
  - ペットタブ内に「前回カルテとの差分ハイライト」（将来拡張）

## 7. 確定事項（2026-04-02）

1. URLは最終想定として`/customers/manage`を採用する
2. β期間のページ名は`顧客管理（β）`を採用する
3. ペットタブ順は`名前順`を採用する
4. カルテ一覧は店舗設定で件数`N`を可変にする
5. 日誌一覧は店舗設定で表示条件（公開済みのみ/下書き含む）を可変にする
6. 編集UIは既存同様にモーダル優先で実装する
7. サイドバーに新メニューを追加し、名称は`顧客管理（β）`とする

## 8. 仕様確定後の実装分解（予定）

- UI: 新規ページ・タブレイアウト・既存モーダル再利用
- API/取得: 顧客単位でのペット/カルテ/日誌取得最適化
- 導線: サイドバー + 既存ページからの相互リンク
- テスト: E2E 2〜3本、必要な単体テスト追加
